## 向前端展示 DeepSeek 思考过程

### 目标

在未来单独的 OpenSpec change 中，将 DeepSeek Chat Completions 返回的
`reasoning_content` 通过后端 API/SSE 安全地传递给前端，并提供清晰的展示与折叠交互。

### 当前变更边界

- 当前 DeepSeek 适配只在后端内部保留并按协议回传 `reasoning_content`。
- 当前不修改现有聊天 API 响应结构、SSE 事件格式或前端界面。
- 当前不向最终用户暴露模型思考内容。

### 后续实现前需要明确

- 区分思考内容、最终回答、工具调用及工具结果，避免消息顺序混乱。
- 设计默认折叠、加载中、空内容、中断、错误和重试状态。
- 明确会话持久化、审计、日志脱敏、权限和历史消息重放规则。
- 增加后端契约测试、流式顺序测试和前端交互测试。

## 多部门归属（用户同时属于多个部门）

### 背景

来自 change `hierarchical-admin-permissions`（部门分级权限模型）的设计决策：该 change
只实现单部门归属，但数据模型上预留了扩展位——`UserRecord` 的部门字段设计为
`department_ids: list[str]`（JSON 数组列），当前业务逻辑强制长度 ≤ 1。

### 后续实现前需要明确

- `_require_manage_scope` 的相等比较逻辑（`actor.department_id == target.department_id`）
  需要改为集合交集判断（多个部门管理员共同管理同一用户时的权限合并规则）。
- 多归属场景下，`department_admin` 创建用户时"新用户归属于哪个部门"的默认规则需要重新设计
  （当前是"强制等于操作者自己的部门"，多归属下可能需要显式选择）。
- 知识空间访问权限（`spaces` 字段）与多部门归属的联动规则（一个用户属于多个部门时，
  是否自动获得这些部门关联的全部知识空间访问权限，还是仍需单独授权）。
- 用户列表/详情查询在 `department_admin` 视角下的可见范围规则（同一用户被多个部门共享时，
  各部门管理员看到的字段是否需要差异化）。
- 当前设计不限制单个部门内 `department_admin` 的数量，多个部门管理员共同管理同一部门时是否需要
  引入协作规则（例如互相不能操作对方账号，见 design.md Decision 4 的管理员账号保护说明）；如需
  数量上限约束，需要在未来的 change 中补充（design.md Open Questions 第 1 项）。
- `dev_header`/`reverse_proxy`/`AUTH_ENABLED=false` 三种鉴权模式下，`AuthContext.department_id`
  恒为 `None`（这些模式没有部门归属的数据源），因此这些模式下配置的 `department_admin` 角色实际
  上管理不了任何用户——这是刻意的安全默认（宁可拒绝也不误放行），不是缺陷，但意味着当前只有
  `local_credentials` 鉴权模式真正支持部门管理员；如果未来需要在其他模式下也支持，需要扩展
  Header/环境变量的角色解析格式以携带部门信息（design.md Open Questions 第 3 项）。

完整背景见 `openspec/changes/hierarchical-admin-permissions/design.md` 的 Open Questions 一节。

## RAG 编排路径扩展位

`pluggable-rag-orchestration-paths` change 引入了"编排路径注册表"机制（`app/agents/rag_graph.py` 的 `build_rag_graph_registry`），当前已交付 `baseline`（原有先检索后回答拓扑）和 `agentic`（先裸答、按需检索）两条路径。未来新增路径（例如 Corrective RAG——检索后按相关性评分决定是否使用上下文；GraphRAG——基于图结构索引的检索）只需在 registry 里新增一个"路径名 → 已编译图"的条目，知识空间的 `rag_path` 字段无需迁移即可设置为新路径名。

已知局限：`agentic` 路径的检索命中判定只做"有没有检索到文档"的二元判断，不做相关性过滤——检索到的文档内容与问题实际不相关时，当前实现仍会当作命中处理并生成基于该上下文的回答。这正是 Corrective RAG 要解决的问题，本次不实现。

另见本 change 的 Open Questions：
`openspec/changes/pluggable-rag-orchestration-paths/design.md`（Open Questions 一节）。

## 检索增强路径：Corrective RAG 与 GraphRAG

### 背景

来自 change `pluggable-rag-orchestration-paths`（可插拔 RAG 编排路径）的设计决策：该 change
把固定的对话编排拓扑改造为"路径注册表"机制，首批只交付 `baseline` 和 `agentic` 两种路径，
但注册表本身设计为可扩展——新增路径只需要在 registry 里加一个条目，知识空间的 `rag_path`
字段不需要因为新增路径而做数据迁移。

### 已知局限（agentic 路径引入）

`agentic` 路径依赖模型的自我认知边界判断"是否需要检索"，这在私有知识库场景下有一个特有的
失败模式：模型对训练数据里见过的通用概念很自信（例如"报销比例一般是多少"），即使用户实际
问的是本组织的私有配置，模型也可能不触发检索、直接给出一个通用但错误的答案。`agentic` 路径
的检索命中判定也只是"有没有检索到文档"的二元判断，不做相关性过滤——检索到的文档即使和问题
不相关，也会被当作上下文使用（见上文「RAG 编排路径扩展位」与 design.md Open Questions）。

### Corrective RAG（纠偏检索）

- 目标：检索永远先跑（向量检索成本低），但增加一个 `grade_documents` 节点，按相关性给检索
  结果分档（相关/部分相关/不相关），而不是只判断"有没有文档"。
- 分档后的处理：相关 → 正常带上下文回答；部分相关 → 查询改写后再检索一次；不相关 → 丢弃上下文
  或据实说明未找到相关内容，避免模型在不相关上下文的误导下给出错误答案。
- 实现成本：中等，新增一个节点 + 一次轻量模型调用（或专用分类器），不需要更换存储后端，可以
  作为 registry 里的新路径 `"corrective"` 独立交付。

### GraphRAG（图结构检索）

- 目标：知识以图结构（实体 + 关系）组织，而不是纯向量相似度切片，适合"实体关系密集"型知识库
  （例如组织架构、人员职责关系类问答）。
- 实现成本：三者中最高，因为换的是数据存储形态——需要新的 ingestion 流程（用 LLM 从文档抽取
  实体和关系）、一个图存储后端（如 Neo4j 或轻量内存图）、一个新的 `RetrieverProvider` 实现
  （图遍历/社区摘要代替相似度检索）。可以与现有向量检索共存做成 Hybrid Retriever。
- 与"部门分级权限"的交互：某个知识空间切换到 GraphRAG 意味着该空间下的文档需要重新跑一遍图
  抽取 ingestion，这是一个隐含的资源消耗决策，不是纯运行时开关——UI 上需要向管理员显式提示。

### 后续实现前需要明确

- `grade_documents` 的分档粒度和判定方式（LLM 打分 vs 专用小模型 vs 基于检索分数阈值）。
- GraphRAG 的图存储后端选型（是否需要引入 Neo4j 等新的基础设施依赖，还是用内存图满足现阶段
  规模）。
- 知识空间粒度下，GraphRAG ingestion 与现有向量 ingestion 流程如何共存（同一批文档是否同时
  维护两套索引，还是切换路径时需要用户显式触发重新处理）。

## 分级角色对应的管理页面

当前 `/admin/projects` 是仅允许 `super_admin` 访问的遗留单一管理面板，
没有覆盖三类角色各自需要的独立工作流。后续变更需要分别设计和实现：

- `super_admin` 管理页面：全局部门、管理员、用户和项目治理。
- `department_admin` 管理页面：仅管理本部门普通用户和授权范围内的知识空间。
- 普通用户页面：只提供本人可访问的业务功能，不暴露管理操作。

三类页面的路由、导航、字段可见性和 API 权限需要在各自的 OpenSpec change
中定义；本次修复不扩大 `/admin/projects` 的访问范围。
