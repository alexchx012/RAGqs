# 风险登记表

本文件记录对当前代码库的只读漏洞与风险审计结果。审计范围覆盖安全漏洞、RAG 质量漏洞、Agent 架构漏洞、工程质量漏洞和可维护性漏洞。本文件不修复代码，只为后续 OpenSpec change 提供事实依据。

## 审计边界

- 本次审计基于静态读取当前工作区文件，未执行真实攻击、真实模型调用、真实 Milvus 写入或生产流量压测。
- `.env` 中确认存在已填写的 `DASHSCOPE_API_KEY`，本文只记录存在性，不记录密钥值。
- 直接的任意命令执行入口未在 `app/` 主应用代码中确认；启动脚本会调用 Docker、Python、PowerShell 原生命令，但参数有显式校验，仍列入部署面风险观察。
- 直接的公开 API 任意文件读取入口未确认；文件读取主要发生在上传后索引和内部 loader 路径，仍需人工确认是否会向外暴露任意本地路径索引能力。

## Open Questions

- `default` knowledge space 的设计语义是什么：全局检索入口，还是普通默认空间？这决定 R-008 是否属于实际越权。
- 生产或试运行环境是否会把 FastAPI 直接暴露到非本机网络？如果会，R-002 和 R-009 严重程度应上调。
- 反向代理是否已经在部署层强制清洗所有外部 `X-RAG-*` headers？如果没有，R-003 是直接权限绕过。
- 当前 `LangChain`/`langgraph` legacy agent 是否有默认 recursion limit 或 max iterations？代码中未显式配置，需要运行时确认。
- Milvus `similarity_search(..., filter=dict)` 在当前 `langchain_milvus` 版本中是否完全按预期执行？需要用真实 Milvus 数据验证 filter 语义。
- 检索 score 是否由当前 Milvus/LangChain 返回并写入 metadata？如果没有，后续 threshold/rerank 需要先补 score 来源。
- 检索审计数据是否属于敏感数据，是否已有保留期限、脱敏策略和访问审计要求？
- HTML 上传是否是业务必需？如果不是，建议先移出默认允许扩展名。
- 是否需要保留 `legacy` agent runtime？如果不需要，应作为单独 change 移除或冻结。
- `.env` 中当前 DashScope key 是否已经对外分享过？需要项目所有者人工确认是否轮换。
